Understanding Base64 Encoding

How it works, when to use it, when not to, and the gotchas that trip people up.

What Is Base64?

Base64 is a binary-to-text encoding scheme that converts binary data into a sequence of printable ASCII characters. It uses 64 characters: A-Z, a-z, 0-9, +, and / (plus = for padding). Every 3 bytes of binary data become 4 characters of Base64 text.

The key insight: Base64 doesn't encrypt or compress data — it encodes it for safe transport through systems that only handle text (email, JSON, URLs, HTML attributes).

How It Works (Simplified)

Base64 takes 3 bytes (24 bits) of input and splits them into four 6-bit groups. Each 6-bit value (0-63) maps to a character in the Base64 alphabet. If the input isn't a multiple of 3 bytes, padding characters (=) are added:

Input:  "Man"  →  ASCII 77, 97, 110
Binary: 01001101 01100001 01101110
Groups: 010011 010110 000101 101110
Base64:  T      W      F      u
Output: "TWFu"

Size overhead: Base64 output is approximately 33% larger than the original binary data (3 bytes → 4 characters). This is the trade-off for text-safe transport.

When to Use Base64

Use CaseWhy Base64 Helps
Data URLsEmbed small images, fonts, or files directly in CSS/HTML as data:image/png;base64,...
JSON API payloadsSafely include binary data (images, files) inside JSON strings without breaking syntax
Email attachments (MIME)SMTP only handles 7-bit ASCII. Base64 encodes binary files for email transport.
JWT tokensJWT header and payload are Base64URL-encoded (with - and _ replacing + and /)
Basic Auth headersHTTP Authorization: Basic base64(username:password)
Storing binary in text-based databasesWhen you must store binary data in a text column

When NOT to Use Base64

  • For large files — The 33% overhead adds up. A 1MB image becomes 1.33MB. Use binary transfer (multipart/form-data) for file uploads instead.
  • For encryption — Base64 is encoding, not encryption. Anyone can decode it. Use AES, RSA, or TLS for security.
  • For compression — Base64 makes data larger, not smaller. Compress first (gzip/brotli), then encode if needed.
  • For storing passwords — Never Base64-encode passwords. Use bcrypt, scrypt, or argon2 for password hashing.

Standard vs URL-Safe Base64

VariantCharacter 62Character 63PaddingUse Case
Standard+/=Email, MIME, PEM certificates
URL-Safe-_(removed)JWT, URL parameters, filenames

URL-safe Base64 replaces + and / because they have special meaning in URLs (+ is a space, / is a path separator). The trailing = padding is typically stripped since it can cause issues in URL parameters.
